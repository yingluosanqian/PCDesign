设计一个命令行工具，帮助一名独立工程师通过让若干 AI agents 围绕一份
不断演进的设计文档迭代，从而收敛到一份经得起推敲的软件设计。

整体机制：一个 agent 担任"作者"——它每一轮都拥有并重写这份设计文档。
另有一个或多个 agent 担任"评审"——它们读取当前文档、提出异议，但不
修改文档。工具把评审们的产出仲裁成一份确定性的、供作者在下一轮直接
消费的决策。循环的终止条件是：设计已经"够好了"，或者再跑一轮不太可
能带来进一步收益。

硬性约束：
- 本地 CLI 工具，不是 service。所有状态落在项目目录下的磁盘上。不
  自建 daemon，也不对外起网络服务。
- Agents 是以子进程方式拉起的外部 CLI（例如 codex、claude）。不要
  假设能直接调用库级 API。作者的上下文必须能在新启动的 CLI 进程里
  被接续。
- 仅靠"某一轮看起来很干净"不足以判定收敛；必须有某种跨轮的稳定性
  信号。
- 当循环没有实质进展时必须能主动停下来——防止用户无意间把 agent
  调用预算烧光。
- 人必须能在两轮之间检视或推翻本轮的决策，且不需要为此再跑一次
  agent。
- 目标用户：一名工程师、一台机器。不考虑多人协作、远程协同、认证。

Rationale（方案论述）部分请从第一原理推导方案。特别要论证：
(i) agent 角色的数量以及彼此的边界；
(ii) 每一个你引入的 per-round 步骤——每一步都带来延时与成本，因此
    每一步都必须有一个理由；
(iii) 收敛判据为什么是你选的那一个，而不是某个更简单的；
(iv) 人工 escape hatch 本身如何避免变成一个新的 divergence 来源。
